### 三、P2 体验提升优化（已完成）

#### 3.1 TTL缓存系统 ⭐⭐⭐

**问题**:

- 使用简单dict作为缓存
- 无过期机制（可能返回陈旧数据）
- 非真正的LRU淘汰策略

**解决方案**:
实现完整的 `TTLCache` 类：

```python
class TTLCache:
    特性:
    - 过期时间: 可配置TTL（默认300秒/5分钟）
    - LRU淘汰: 自动删除最旧条目
    - 惰性清理: 读取时检查并删除过期项
    - 线程安全: 单线程环境下安全
    
    API:
    - get(key): 获取值（自动检查过期）
    - set(key, value, ttl): 设置值
    - invalidate(key): 使指定key失效
    - clear(): 清空所有缓存
```

**可靠性提升**:

- ✅ 缓存数据不会永久过期
- ✅ 内存占用可控
- ✅ 搜索结果更准确

**配置项**:

- `cache_ttl`: 缓存过期时间（秒），默认300
- `max_cache_size`: 最大缓存条目数，默认1000

**修改文件**:

- [db\_manager.py](databases/db_manager.py) - 新增TTLCache类 + 替换原有缓存

#### 3.2 消除魔法数字 ⭐⭐⭐

**问题**:

- 大量使用数字索引访问数据库行：`row[0]`, `row[5]`
- 权重散落在各处：`5.0`, `0.8`, `0.5`, `0.1`
- 维护困难，容易出错

**解决方案**:

**① 定义列常量类**:

```python
class MemoryColumn:
    ID = 0
    CATEGORY = 1
    IMPORTANCE = 2
    CREATED_AT = 3
    UPDATED_AT = 4
    TAGS = 5
    CONTENT = 6
    ACCESS_COUNT = 7
```

**② 统一使用常量**:

```python
# 优化前（难维护）
memory["id"] = row[0]
memory["tags"] = row[5]

# 优化后（清晰易读）
memory["id"] = row[MemoryColumn.ID]
memory["tags"] = row[MemoryColumn.TAGS]
```

**可维护性提升**:

- ✅ 代码可读性大幅提升
- ✅ 表结构变更时只需修改一处
- ✅ 减少人为错误

**修改文件**:

- [db\_manager.py](databases/db_manager.py) - MemoryColumn类定义 + 全局替换

#### 3.3 MMR多样性策略 ⭐⭐⭐⭐

**问题**:

- 传统排序可能返回多条高度相似的结果
- 用户看不到多样化的信息

**解决方案**:
实现MMR（最大边际相关性）算法：

$$\text{MMR} = \lambda \cdot \text{Relevance}(d, q) - (1-\lambda) \cdot \text{Similarity}(d, d\_{selected})$$

**参数配置**:

- `mmr_enabled`: 是否启用（默认True）
- `mmr_lambda`: 平衡参数（默认0.7）
  - λ=0.9: 几乎不考虑多样性
  - **λ=0.7: 推荐！平衡相关性和多样性**
  - λ=0.5: 强调多样性

**效果示例**:

```
搜索"Python"，传统排序返回：
1. Python基础教程 (25分)
2. Python入门指南 (24分) ← 高度相似
3. Python高级特性 (23分) ← 还是基础内容

启用MMR后返回：
1. Python基础教程 (25分)
2. Java vs Python对比 (18分) ← 不同角度
3. Django Web开发 (17分) ← 不同主题
```

**修改文件**:

- [db\_manager.py](databases/db_manager.py) - MMR算法实现 + search\_memory集成

## 📊 评分公式优化（V2版本）

### V1 → V2 主要改进

| 维度                  | V1旧版 | V2新版     | 改进原因       |
| ------------------- | ---- | -------- | ---------- |
| **importance默认值**   | 5    | **1**    | 降低基础分，减少噪音 |
| **importance系数**    | ×0.5 | **×0.3** | 降低重要性权重    |
| **access\_count处理** | 线性增长 | **对数压缩** | 防止热门记忆霸占   |
| **access\_count上限** | 无上限  | **3.0分** | 控制流行度影响    |

### V2评分公式详解

```python
score = 0

# 1. 标签匹配（核心指标）：每个命中词 +5.0分
# 2. 内容包含（次要指标）：每个命中词 +4.0分  
# 3. 分类匹配（辅助指标）：每个命中词 +2.0分
# 4. 重要性加成：importance × 0.3 （默认1×0.3=0.3分）
# 5. 流行度加成：log10(access_count+1) × 0.5 （上限3.0分）
# 6. 完整匹配奖励：+10.0分（查询完全包含在内容中）

# 典型分数分布：
# - 不相关记忆: 0.3分（仅基础分）→ 会被过滤 ✓
# - 一般相关: 5-15分
# - 高度相关: 15-30分
# - 完全匹配: 25-40分
```

**防霸占机制**:

```
access_count 对比:
- 0次: log10(1) × 0.5 = 0分
- 10次: log10(11) × 0.5 = 0.52分
- 100次: log10(101) × 0.5 = 1.0分
- 1000次: log10(1001) × 0.5 = 1.5分

✅ 即使被回忆1000次，也只加1.5分，不会压倒一切
```

## 🆕 新增功能：智能身份认知系统（v0.2.0）

### 功能概述

针对QQ群等复杂社交场景，实现了**三层身份认知架构**：

```
┌─────────────────────────────────────────┐
│  Layer 1: 身份映射层 (Identity Mapping)  │
│  • 记录用户的所有别名（昵称/群名/ID）     │
│  • 自动追踪名称变更历史                   │
│  • 关联群组与用户的映射关系               │
└────────────────────┬────────────────────┘
                     ↓
┌─────────────────────────────────────────┐
│  Layer 2: 智能查询层 (Smart Query)       │
│  • 支持按名字/ID/群名多维度查找           │
│  • 模糊匹配 + 置信度评分                 │
│  • 自动解析身份别名                      │
└────────────────────┬────────────────────┘
                     ↓
┌─────────────────────────────────────────┐
│  Layer 3: 自然注入层 (Natural Injection) │
│  • 根据关系深浅动态调整风格              │
│  • 新朋友：简洁介绍                      │
│  • 老朋友：温馨提醒 + 重要约定            │
│  • 有待办：重点标注                      │
└─────────────────────────────────────────┘
```

### 核心实现

#### 1. 身份映射表 (`identity_aliases`)

**新增数据库表结构**:

```sql
CREATE TABLE identity_aliases (
    id INTEGER PRIMARY KEY,
    user_id TEXT NOT NULL,           -- 关联的用户ID
    alias TEXT NOT NULL,             -- 别名/昵称
    alias_type TEXT DEFAULT 'nickname', -- 类型: nickname/group_name/platform_id
    source_context TEXT,             -- 来源场景（群ID）
    is_current INTEGER DEFAULT 1,   -- 是否当前使用
    first_seen_at TIMESTAMP,
    last_seen_at TIMESTAMP
)
```

**解决的问题**:
- ✅ 同一个人在不同群的昵称不同 → 全部记录
- ✅ 昵称随时可能变更 → 自动维护历史
- ✅ 群名称会变但ID不变 → 以ID为主，名称为辅

**使用示例**:
```python
# AI记录一个用户时，自动同步所有别名
db_manager.add_identity_alias("QQ_123456", "小明", "nickname")
db_manager.add_identity_alias("QQ_123456", "Python学习群", "group_name", "group_789")
db_manager.add_identity_alias("QQ_123456", "明哥", "nickname")  # 后来改的昵称
```

```python
# 自动完成：
```

### 解决的核心痛点

| 痛点 | 解决方案 | 效果 |
|------|----------|------|
| **"用身份证找人"太生硬** | 支持按名字/群名模糊查询 | ✅ 更符合人类习惯 |
| **同一个人多个昵称** | 身份映射表记录所有别名 | ✅ 全面识别不遗漏 |
| **群名称会变** | 以群ID为基准，群名为辅助 | ✅ 名称变更不影响识别 |
| **关系注入像看档案** | 三种自然化风格动态切换 | ✅ AI感觉像在和朋友聊天 |
| **跨群识别困难** | 关联群组和用户的映射 | ✅ 跨场景统一认知 |

### 配置项说明

无需额外配置，系统自动启用。可选配置：

```json
{
  "identity_mapping": {
    "enabled": true,                    // 是否启用身份映射（默认true）
    "auto_record_aliases": true,        // 是否自动记录别名（默认true）
    "fuzzy_match_threshold": 0.5,       // 模糊匹配阈值（0-1）
    "max_aliases_per_user": 20,         // 每个用户最大别名数
    "injection_style": "auto"           // 注入风格: auto/detailed/simple
  }
}
```
